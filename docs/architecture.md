## System Overview

**Repo**: pnpm 10.28.2 + Turborepo monorepo (`pnpm-workspace.yaml`: `apps/*`, `packages/*`). One Next.js 16 App Router app ŌĆö `apps/DocFlow` (React 19, Tiptap v3, Tailwind v4, shadcn/ui new-york + Radix, TanStack Query v5, Zustand v5) ŌĆö plus three vite-built Tiptap extension packages under the `@syncflow` scope (`packages/alert`, `packages/bilibili`, `packages/transformer`), all effectively vestigial (see Removal Candidates). Fork of `xun082/DocFlow` (origin `javaxhaskell/intent-writing-studio`, branch `feat/m0-discovery`); root `LICENSE.md` (MIT, "Copyright (c) 2025 Moment") is the attribution surface to preserve.

**Frontend-only**: the only in-tree API route is `apps/DocFlow/src/app/api/health/route.ts` (GET, load-bearing for Docker HEALTHCHECK). Zero server actions. Every feature calls an **external NestJS backend** through `NEXT_PUBLIC_SERVER_URL` ŌĆö and the **git-tracked** `apps/DocFlow/.env.development` / `.env.production` pin it to `https://api.codecrack.cn` (the upstream author's production host), plus `wss://api.codecrack.cn/collaboration` (Yjs), `https://note.codecrack.cn` (socket.io), and a hardcoded `wss://flow.codecrack.cn` in the workflow canvas. Every dev run and prod build currently targets a third party's live infrastructure.

**Product target** (`docs/build-prompt.md`, ADR 0002 in `docs/`): intent-driven, multi-pathway AI writing studio on **Supabase** (Postgres/RLS/pgvector/Edge Functions or Next route handlers) with an env-pinned LLM (`LLM_PROVIDER`/`LLM_MODEL`/`ANTHROPIC_API_KEY`). Current state of that target: **zero scaffolding** ŌĆö no `@supabase/supabase-js`, no `supabase/` dir or migrations, no AI SDK, no ModelGateway. Root `.env.local` stages the Supabase/LLM credentials, but Next.js only loads env from the app directory, so **the app cannot see any of them today** (`apps/DocFlow/.env.local` does not exist). `docs/`, `CLAUDE.md`, `.env.example`, and `.greptile/` are all **untracked** working-tree files.

Verified build: `fnm exec --using=24 -- pnpm build` exits 0 with no backend (all backend fetch failures are swallowed into `RequestResult`), but system Node 22 fails engine-strict install ŌĆö every invocation must pin Node 24 / pnpm 10.28.2.

## Editor

Assembled once by `ExtensionKit({provider})` in `apps/DocFlow/src/extensions/extension-kit.ts` (~40 Tiptap v3 extensions; barrel: `src/extensions/index.ts`), instantiated in `src/app/docs/[room]/page.tsx` via `useEditor` with `immediatelyRender:false` and **`shouldRerenderOnTransaction:false`** (any attr-badge UI must subscribe to editor transactions itself). Live editor instance is shared to out-of-tree panels via `src/stores/editorStore.ts` (Zustand).

**Persistence is Yjs-first, never HTML/JSON at rest**: Y.Doc field `content` via `@tiptap/extension-collaboration` + `CollaborationCaret`; `y-indexeddb` (`tiptap-collaborative-{documentId}`) restores first, then `HocuspocusProvider` connects (`src/hooks/useCollaboration.ts`). History snapshots are opaque `Y.encodeStateAsUpdate` binaries in IndexedDB `docflow-snapshots` (`src/services/snapshot`, 5-min auto-snapshot via `useEditorHistory`). New node attrs persist automatically once in the schema; `renderHTML`/`parseHTML` matter only for clipboard and HTML-export surfaces.

**Block identity**: `@tiptap/extension-unique-id` at `extension-kit.ts:88-91` ŌĆö `types: ['paragraph','heading','blockquote','codeBlock','table']`, `filterTransaction: !isChangeOrigin(tr)` (only the originating client assigns ids), package defaults otherwise (attributeName `id`, uuidv4, rendered/parsed as `data-id`, duplicate regeneration on paste/split for its own attr only). **The agent anchor contract already treats this as blockId**: `useDocumentEdit.ts` resolves `AgentOp.nodeId`/`AgentAnchor.anchorNodeId` by scanning `doc.descendants` for `node.attrs.id`. Coverage gap: lists, taskItems, imageBlock, details, horizontalRule, draggableBlock get **no** ids.

**AI diff mechanism**: custom Mark `agentSuggestion` (`src/extensions/AgentSuggestion/index.ts`; attrs `suggestionId` + `variant` added|deleted ŌåÆ `data-*`). `useDocumentEdit.ts` streams intentŌåÆanchorŌåÆproposal (SSE via `src/services/collaboration/index.ts`), injects marks into proposal content, replace-ops mark the old block `deleted` in place and insert the marked replacement after it; `acceptAgentSuggestion`/`rejectAgentSuggestion` delete the losing variant's **whole parent block** and strip winning marks ŌĆö so accept destroys the old block's id/metadata, and one logical block transiently exists as two. Review UI: `src/app/docs/_components/AgentEditPanel`. Marks are written into the shared Yjs doc ŌĆö pending diffs broadcast to all collaborators and survive tab close with no cleanup path.

**Other editor systems**: two independent drag systems (`@tiptap/extension-drag-handle-react` wrapped by `ContentItemMenu` with per-node actions via `useContentItemActions`; custom `DragHandler` extension for sidebar palette drops via `application/x-block-type`). `DraggableBlock` wrapper node (blockType attr + React node view) is registered but unused by any flow. Slash menu via `@tiptap/suggestion` (`src/extensions/SlashCommand`). `MarkdownPaste` (mdast/micromark GFM parse + a custom clipboard markdown serializer that **omits all attrs** ŌĆö plain-text copies already satisfy the no-provenance-in-copies rule). `JsonPaste` re-inserts full `editor.getJSON()` from the `text/json` clipboard channel populated by the copy handler in `[room]/page.tsx` ŌĆö **this path carries all node attrs verbatim across documents**. Headless converter `src/utils/markdown-to-tiptap.ts` reuses ExtensionKit (new extensions inherited automatically; produced nodes have null ids until inserted live).

**Patterns to copy** for new block metadata: `addGlobalAttributes` across a types list (`FontSize.ts`, UniqueID itself), Mark/Node `data-*` parse/render (`AgentSuggestion`, `Comment`, `DraggableBlock`), Commands-module augmentation + `tr.setNodeMarkup` (`AgentSuggestion` commands). Read-only mode toggles `editor.setEditable` without recreating the editor ŌĆö metadata-mutating commands must check editability.

## AI Clients

All AI is frontend-only against the absent backend. Uniform transport: `clientRequest.streamPost` (`src/services/request/client.ts`) ŌĆö fetch POST + `ReadableStream` reader, newline split, `parseOpenAIStreamLine` (`data: {json}` prefix, `[DONE]` ignored, bare-JSON fallback). No WebSocket/EventSource for AI. Three primitives exist (`sse` raw-Response, `sseStream` via `@azure/core-sse` ŌĆö dead code, `streamPost`), all with 401-refresh-and-retry (which can double-execute non-idempotent generations).

1. **ChatAiApi** (`src/services/chat-ai/index.ts`): streaming `/api/v1/chat/completions|autocomplete|polish|brainstorm` + conversation CRUD + models. Completions returns new conversation id via the **`Session-Id` response header**. Brainstorm demuxes n=1ŌĆō5 concurrent results by `choices[].index` ŌĆö implemented but **zero UI call sites** (orphaned `styles/partials/ai-brainstorm.css`); Autocomplete/Polish likewise unwired. Consumed by `app/chat-ai` (useChat, ChatAIPanels).
2. **streamDocumentEdit** (`src/services/collaboration/index.ts`): POST `/api/v1/collaboration/agent/edit`, 120s timeout, line-delimited SSE `{type,data}` ŌĆö `thinking_token|intent_token|anchor_token|proposal_token|intent|anchor|proposal|error|done`; proposal ops `insert_after|insert_before|replace|append_to_doc` with `nodeId` + Tiptap content. A 3-node LangGraph pipeline (intentŌåÆanchorŌåÆproposal) ŌĆö **the direct forerunner of the intent-graph/proposal system**. Consumed by `useDocumentEdit` ŌåÆ `AgentEditPanel`. Unknown/hallucinated nodeIds silently fall back to append-at-document-end.
3. **AiApi legacy** (`src/services/ai/index.ts`): continuations/knowledge-QA/assistant/images/podcasts ŌĆö only the knowledge CRUD subset has UI call sites (dashboard knowledge pages; duplicated by `src/services/knowledge`).

**Thinking/reasoning surface** (slated for removal ŌĆö wide blast radius): `thinking_token` in the AgentEditStreamEvent union; `useDocumentEdit` accumulators; AgentEditPanel ThinkingBlock/StreamingBlock/HistoryRound; ChatAIPanels µĘ▒Õ║”µĆØĶĆā collapsible + deep-thinking toggle; `useChat` reasoningContent buffering; `enable_thinking`/`thinking_budget`/`support_thinking`/`reasoning_content` across `chat-ai/type.ts`, `chat-ai/types.ts`, `constants.ts` (default enableThinking:true/4096).

## Collaboration

Three independent realtime channels, all expecting absent external backends:

1. **Doc collaboration** ŌĆö `src/hooks/useCollaboration.ts`. Boot sequence: HTTP `GET /api/v1/documents/{id}/permissions` (gate; `useDocumentPermission.ts`) ŌåÆ Y.Doc creation ŌåÆ IndexeddbPersistence `synced` ŌåÆ HocuspocusProvider (`NEXT_PUBLIC_WEBSOCKET_URL`, room name = raw `[room]` param, `token` = `auth_token` cookie JWT). Awareness field `user` `{id,name,color,avatar}` (deterministic color hash). Custom protocol extension: server must push stateless JSON `{type:'server:permission', readOnly}` which overrides HTTP-derived read-only state. **If the env var is unset the app silently degrades to IndexedDB-only editing** (provider null, isServerSynced forced true) ŌĆö the fork's current de facto mode; per-browser copies will diverge and CRDT-merge unpredictably when a real backend appears. `onAuthenticationFailed` only toasts and lets the user keep editing a stale local cache. Read-only is advisory client-side (`setEditable`) ŌĆö a real backend must reject writes server-side. Reconnection is 100% library default.
2. **Workflow canvas** ŌĆö `app/dashboard/workflow/_components/canvas.tsx:133`: second HocuspocusProvider **hardcoded to `wss://flow.codecrack.cn`**, single unauthenticated global room `flow-room`, Y.Maps `nodes`/`edges`, mouse-cursor awareness. Any visitor worldwide shares the room.
3. **Notifications** ŌĆö `src/hooks/ws/useNotificationSocket.ts`: socket.io to `NEXT_PUBLIC_NOTIFICATION_WEBSOCKET_URL`, **JWT in both handshake auth and URL query string** (leaks into logs), `forceNew:true` per connect (duplicate-manager risk). Events: `connected`, `online_users`, `user_status_change`, `podcast_event`; emits `get_online_users`, `ping`. Mounted via `NotificationSocketProvider` in the dashboard layout.

Adjacent: LiveKit A/V rooms (`src/services/rooms` ŌĆö backend mints token + returns the LiveKit WS URL; separate `NEXT_PUBLIC_LK_RECORD_ENDPOINT` recording service). Dead deps declared but never imported: `@hocuspocus/server`, `y-webrtc`, `livekit-server-sdk`. Hocuspocus v3 wire protocol must be matched when rebuilding the WS server.

## Data Layer and Expected Backend API

No axios: two hand-rolled fetch wrappers in `src/services/request/` ŌĆö `ClientRequest` (`client.ts`, ~1300 lines: Bearer from non-HttpOnly `auth_token` cookie, automatic 401 ŌåÆ POST `/api/v1/auth/refresh` with `refresh_token` cookie (queued replay, max 2 attempts, then clear cookies + hard redirect `/auth`), Promise.race timeouts (never applied to streaming body reads), retry loop, Sentry breadcrumbs, three streaming primitives) and `ServerRequest` (`server.ts`, unauthenticated RSC variant). **Every response must be the business envelope `{code, message, data, timestamp}`** (code 0 or 2xx = success); services return `RequestResult<T> = {data, error, status}` instead of throwing, with per-call errorHandler maps. Many call sites hard-check `data.code !== 200`. Raw PostgREST responses will be rejected unless an envelope adapter is kept.

**Auth**: email+code, email+password, registration, and backend-initiated GitHub OAuth (full-page redirect to `${SERVER_URL}/api/v1/auth/github?state=...`; `/auth/callback` accepts **tokens as URL query params** and uses the decoded `state` as an arbitrary `router.push` target ŌĆö open redirect). Tokens live in plain JS cookies (`src/utils/auth/cookie.ts`: auth_token, refresh_token, expires_in, refresh_expires_in, auth_timestamp) read directly by ClientRequest, HocuspocusProvider, and socket.io ŌĆö the custody conflict to resolve when Supabase Auth takes over. `src/proxy.ts` (Next 16 middleware) gates `/docs,/dashboard,/chat-ai,/rooms` on cookie presence + a client-set expiry check (with **inconsistent expires_in units** vs cookie.ts); no server-side validation.

**State**: react-query owns server state (global queryClient: 5-min staleTime, 401/403 never retried, global mutation-401 ŌåÆ `/auth`); Zustand owns UI state ŌĆö except `fileStore.ts` which fetches via DocumentApi/organizationService directly. User profile: react-query ŌåÆ localStorage `cached_user_profile` placeholder ŌåÆ network.

**Endpoint surface**: ~120 first-party REST endpoints across `src/services/*` (auth, documents incl. rich share-link/permission model NONEŌåÆFULL, users, chunked upload, comments keyed by Tiptap mark_id, organizations ├Ś21, friend/contacts, notifications ├Ś14 types, chat, collaboration-agent, legacy ai/knowledge, blog, template, trace beacon, livekit, image EXIF, video pipeline, podcast) + 3 WS channels ŌĆö full audited list with dispositions in the endpoint inventory. **Dual write path for content**: Yjs binary state (canonical in practice) AND `PUT /documents/:id/content` Tiptap JSON ŌĆö a Supabase backend must pick one canonical store. Uploads are a custom chunk-merge protocol (check-by-hash, chunk, complete) that has no Supabase equivalent ŌĆö redesign on Storage TUS resumable uploads; uploaded asset URLs are **embedded inside Yjs content** and will 404 when codecrack.cn is cut off; `next.config.ts` has no `images.remotePatterns`. IDs are numeric throughout DTOs vs Supabase Auth UUIDs ŌĆö mapping layer needed.

**Supabase seam**: `src/services/*/index.ts` module boundary ŌĆö preserve each function signature and the RequestResult envelope (via adapter or route handlers under `src/app/api/*` reproducing the envelope) and zero call sites change. Streaming AI endpoints become Next route handlers behind a ModelGateway keeping the OpenAI-chunk wire format (`streamPost` then needs no changes; preserve the Session-Id header contract).

## Top risks (ranked)

1. 1. Live coupling to upstream's production infrastructure: git-tracked apps/DocFlow/.env.development and .env.production point every REST call, the Yjs websocket, and the notification socket at api.codecrack.cn / note.codecrack.cn, and canvas.tsx hardcodes an unauthenticated global room on wss://flow.codecrack.cn ŌĆö every dev run, prod build, and any real user sends traffic, JWTs, and document data to a third party until these are scrubbed.
2. 2. Secret and telemetry leakage: a plaintext Tiptap Pro registry token is committed in .github/workflows/preview.yml; the upstream author's Sentry DSN is hardcoded in three configs with the client init running in ALL environments (sendDefaultPii:true, tracesSampleRate 1, session replay) so the fork's errors and replays exfiltrate upstream; the socket.io client puts the JWT in the URL query string; root .env.local holds live Supabase service-role and Anthropic keys (gitignored, but adjacent to all of the above ŌĆö rotate if ever committed).
3. 3. The backend does not exist: all ~120 /api/v1/* endpoints and 3 WS channels target NEXT_PUBLIC_SERVER_URL with only /api/health in-tree; there is zero Supabase scaffolding (no supabase-js, no migrations, no route handlers, no ModelGateway), so every CLAUDE.md non-negotiable ŌĆö RLS + negative tests, Zod-validated model output, content-hash-guarded regeneration, proposal/audit tables ŌĆö is unstarted, and the entire endpoint inventory is scope to build or cut.
4. 4. Auth bootstrap hard-blocks all editor feature work: proxy.ts cookie-gates /docs and /dashboard, and /docs/[room] refuses to construct the Y.Doc without a successful GET /documents/:id/permissions ŌĆö so intent-graph/provenance/regeneration work cannot be exercised end-to-end until Supabase Auth plus a permission endpoint (or an explicit dev bypass) exists; the existing flow also has token-in-URL callback params and a state-param open redirect that must not survive the port.
5. 5. Deep contract coupling in the data layer: the {code,message,data,timestamp} envelope (code 0/2xx = success) is enforced by ClientRequest and hard-checked at ~30 call sites; the hand-rolled 401-refresh (POST /auth/refresh, 2 attempts, hard redirect, re-issued POST bodies that can double-execute AI generations) fires implicitly on every request including streams; and non-HttpOnly auth_token cookies are read directly by ClientRequest, HocuspocusProvider, and socket.io ŌĆö Supabase session custody must be reconciled with all three or the app 401-loops silently.
6. 6. Block identity coverage gap: UniqueID assigns ids only to paragraph/heading/blockquote/codeBlock/table ŌĆö lists, taskItems, imageBlock, details, etc. have no stable identity, so agent ops targeting them silently fall back to append-at-document-end and intent links for those blocks are impossible; widening the types list is a schema-visible change affecting existing Yjs documents and must be shared with the new BlockMetadata extension via one constant.
7. 7. Persistence integrity hazards: content has two write paths (canonical Yjs binary vs PUT /documents/:id/content JSON) with no declared source of truth; with no WS URL the app silently runs offline-only so per-browser IndexedDB copies diverge and CRDT-merge unpredictably when a backend appears; local snapshots restore old attr states (resurrecting stale intentIds/freshness the server no longer agrees with); and Yjs node attrs merge last-writer-wins, so lock/freshness/content-hash stale-write protection must live server-side.
8. 8. Provenance/metadata leakage through copy paths: the copy handler writes full editor.getJSON() (all attrs) to a text/json clipboard channel that JsonPaste re-inserts verbatim into other documents, HTML paste carries data-* rendered attrs, ContentItemMenu Duplicate clones all attrs, and DOCX/PDF/blog exports consume getJSON() ŌĆö the spec's 'no provenance in copies/exports' rule requires scrubbing at each seam (only MarkdownPaste's plain-text serializer is already safe).
9. 9. CI and test vacuum with hazardous automation: both PR workflows are dead (Node 20 vs engine-strict >=24, nonexistent lint:ci script, wrong Netlify deploy dir), zero test files or runner configs exist so `pnpm test` passes vacuously (masking the mandated RLS negative tests), commitlint is enforced nowhere, and upstream's cron SSH deploy + Docker Hub push workflows remain live in the fork; turbo's build env allowlist also omits all server-side vars, risking stale cached builds once route handlers consume them.
10. 10. Editor state traps for the new metadata layer: shouldRerenderOnTransaction:false means lock/freshness UI won't update on attr-only transactions without explicit subscriptions; any auto-maintenance plugin missing the !isChangeOrigin guard will fight remote Yjs transactions and diverge across peers; acceptAgentSuggestion deletes the old block (destroying its id/metadata instead of transferring it); default keepOnSplit duplicates custom attrs onto both halves of an Enter-split; and setEditable-based read-only mode won't stop metadata commands from dispatching for VIEW/COMMENT users unless they check editability.
